# Перенос FinGam CRM на уровень AlfaCRM

## Context

Школа финансовой грамотности переезжает с **AlfaCRM** (`bmk.s20.online`) на собственную FinGam CRM.
К проекту подключился специалист, который вёл клиентов в AlfaCRM — он второй пользователь и источник требований.

Разрыв между системами огромный. AlfaCRM — это учебный центр целиком: 86 клиентов, 168 уроков,
1461 финансовая операция, 11 тарифов, 5 групп, воронка из 6 этапов со 143 лидами.
Наша CRM — «электронная тетрадь»: клиенты, платежи, расходы. Нет ни расписания, ни групп,
ни педагогов, ни посещаемости, ни абонементов, ни лидов, ни касс.

Цель: пошагово закрыть разрыв, не ломая то, что уже работает, и в конце перенести исторические данные.

## Что обнаружено в AlfaCRM (результат обследования)

### Ключевая идея, которой у нас нет

**Остаток клиента двойной: в деньгах и в уроках одновременно** — «1 320 000,00 / 4 уроков».
Клиент покупает **абонемент** (пакет уроков). Когда урок проводится, списывается один урок
из пакета и его стоимость из денег. Наш `type:'session'` — лишь бледная тень этого механизма.

### Разделы и их содержимое

**Клиенты** (86). Список: ФИО, возраст + дата рождения, общий остаток (деньги / уроки),
статус обучения, контакты, примечание, дата след. посещения.
Вкладки: Текущие, Я — ответственный, Без группы, С группой, Д/р в этом месяце, Архивные.

Карточка ученика: остаток (уроки + деньги), ID, счётчики «Платежи N шт», «Уроки план/факт»,
«Академ. часы план/факт», педагог, **заказчик** (мама как плательщик, отдельно от ребёнка),
мобильный, **счета и абонементы** (Пакет 8, предмет «Финансовая грамотность», период
26.06.2026–03.09.2026), архивные абонементы, скидки, **аллергии/особенности**, номер школы,
группы (+ архивные), регулярные уроки, предмет и уровень, договоры, загруженные файлы,
**виджет посещений** (цветные плитки по датам), единая лента «Уроки и оплаты».

Действия: Добавить, Править, В архив, Отчёты, Приостановки, Управление скидками,
**История изменений**, **Импорт из файла**, **Экспорт в файл**.

**Уроки** (168). Календарь (день/неделя/месяц) с аудиториями + список.
Статусы: **Проведён / Отменён / Запланирован**. Действия: Подтвердить, Отменить,
Вернуть в запланированные, Удалить.
Поля урока: дата, время (11:00–16:00), длительность (300 минут), участники (группа + поимённо),
педагог, тема. Урок бывает групповой и индивидуальный.
Проведение урока порождает списание: «Групповой — Финансовая грамотность, 300м, Посещение, 330 000».

**Группы** (5). Название, период обучения (22.06.2026 — 01.09.2026), клиенты (0/12 — лимит мест),
расписание (ПН 11:00), архивные группы.

**Абонементы** (11). Шаблоны пакетов: «Пакет 4 (1 300 000,00 за 4 шт)», «Пакет 8 (2 400 000 за 8 шт)»,
варианты «с питанием» и «без питания». Поля: название, тарификация (поурочный),
стоимость, «используется» (сколько выдано / активно), дата добавления. Есть архивные.

**Финансы** (1461 операция). Полноценная касса.
Типы: **Доход #N / Расход #N / Выплата ЗП #N**.
Поля: дата, тип, сумма, **счёт (касса)** — «Xolis (карта)», «Наличные (xolis)», «Карта»;
**статья** — «Оплата за занятие», «Налог 1% ИП», «Зарплата тренера», «Доп услуги»;
назначение (клиент → базовый счёт), **плательщик** («Мохинур мама @mmd02_05»), комментарий.

**Лиды** (143). Канбан-воронка, 6 этапов:
Не разобрано (0) → Установлен контакт (113) → Думают/не отвечают (18) → Назначено пробное (1)
→ Проведено пробное (0) → Получена оплата (11). Отдельно «Архив (отказы)».
Карточка лида: имя (обычно «мама такого-то»), дата, примечание, источник
(Реклама в инстаграм / Рекомендация), контакт (телефон + telegram), ответственный.

**Педагоги**, **Задачи**, **Юр. лица**, **Аналитика** — существуют, в этот заход не изучались.

### Визуальный язык: цвета несут смысл

Виджет посещений в карточке ученика кодирует **12 состояний** плитками с датой.
Правило простое и его надо сохранить: **цвет = деньги, значок = факт**.

| Состояние | Фон | Значок |
|---|---|---|
| Запланирован | серый | — |
| Запланирован бесплатный | жёлтый | — |
| Запланирован без абонемента | белый, пунктирная серая рамка | — |
| Предоплачен | зелёный | — |
| Проведён и оплачен | зелёный | ✓ зелёная |
| Проведён в долг | розовый | ✓ розовая |
| Пропуск в долг | розовый | ✗ красный |
| Бесплатный пропуск | жёлтый | ✗ оранжевый |
| Пропуск оплачен | жёлтый | ✗ красный |
| Забыли провести? | белый, красная пунктирная рамка | ? |
| Приостановлен | серый, текст зачёркнут | ⏸ |
| Отменён | серый, текст зачёркнут | ⊖ |

Зелёный — оплачено. Розовый — долг. Жёлтый — бесплатно. Серый — не состоялось.
Зачёркивание — урок не идёт в счёт. Пунктир — требует внимания администратора.

Та же логика в остальных разделах: в списке уроков строки проведённых уроков подсвечены розовым,
заголовки этапов воронки окрашены по мере продвижения к оплате, статус клиента — серый бейдж «Активен».

**Экраны и кликабельность (проверено):**
- Клик по ФИО в списке клиентов открывает карточку (`/customer/view?id=231`), ФИО — ссылка целиком.
- Кликабельны: педагог, заказчик, телефон, абонемент (карандаш для правки), счётчики
  «Платежи 4 шт» и «Уроки п13/ф13» — ведут в отфильтрованные списки.
- Карточка двухколоночная: слева лента событий и виджет посещений, справа узкая колонка-сводка
  (остаток, ID, педагог, заказчик, абонементы, скидки, аллергии, группы, договоры).
- Списки: пагинация 20 строк, «Настройка полей» — пользователь сам выбирает колонки.
- Наверху каждого раздела одинаковая панель действий: Добавить, Править, Отчёты, Другое, Фильтр.

При реализации: снять точные размеры и отступы карточек с живых экранов, не изобретать свои.

### Карточка урока (клик по плитке посещения)

Открывается компактный поповер прямо поверх карточки клиента. Заголовок — «групповой проведен»
(тип + статус). Позиции:

| Поле | Пример |
|---|---|
| Тип | Групповой (пометка «норм») |
| Время | с 11:00 до 14:00 (180 мин.) |
| Аудитория | *(не задан)* — красным, если пусто |
| Педагог(и) | Гришанович Алексей Артурович — ссылка |
| Предмет | Финансовая грамотность |
| Группы | Группа сб 11 — ссылка |
| Список учеников | 7 строк: значок статуса, ФИО (ссылка), **сумма списания** |

Кнопки внизу: **Открыть**, **Провести** (гаснет, если урок уже проведён), **Вернуть**.

**Самое важное открытие.** У каждого ученика в одном и том же уроке **своя стоимость**:
Камилла 335 000, Ахмедов 250 000, Бендик 290 000, Исматова 380 000, Махмудов 250 000,
София 380 000, Эшимов 300 000.

Цена урока не хранится в уроке. Она вычисляется из абонемента конкретного ученика:
`стоимость пакета ÷ число уроков в пакете`, с учётом скидок и того, входит ли питание.
Отсюда и разные пакеты в справочнике («Пакет 4 за 1 300 000» → 325 000 за урок;
«Пакет 4 без питания за 1 100 000» → 275 000 за урок).

Это меняет модель: в `lessons.attendance[].amountCharged` мы **фиксируем** сумму на момент проведения
(чтобы позднее изменение тарифа не переписало историю), но **вычисляем** её из активного абонемента ученика.

## Целевая модель данных (Firestore)

Существующие коллекции `clients`, `payments`, `expenses` сохраняются; `payments` со временем
поглощается коллекцией `transactions`.

```
teachers      { name, phone, telegram, subjects[], rate, active }
subjects      { name, levels[] }                       // «Финансовая грамотность»
packages      { name, lessonsCount, price, active, archived }
                                                       // шаблон: «Пакет 8 (2 400 000 за 8)»
services      { name, price, perLesson, active }       // доп. услуги: питание и т.п.
groups        { name, subjectId, teacherId, capacity,
                schedule[{ weekday, time, durationMin }],
                startDate, endDate, studentIds[], archived }

subscriptions { clientId, packageId, name, subjectId,
                lessonsTotal, lessonsUsed, price,
                serviceIds[],                          // подключённые доп. услуги (питание)
                startDate, endDate, status }           // выданный клиенту абонемент

lessons       { date, startTime, durationMin, type: group|individual|trial,
                groupId, teacherId, subjectId, topic,
                status: planned|conducted|cancelled,
                attendance: [{ clientId, status: present|absent|excused,
                               subscriptionId, amountCharged, servicesCharged }] }

leads         { childName, childAge, parentName, phone, telegram,
                source, stage, note, responsibleId,
                createdAt, stageChangedAt, archivedReason }

accounts      { name, kind: cash|card }                // кассы: «Xolis (карта)», «Наличные»
categories    { name, kind: income|expense }           // статьи: «Оплата за занятие», «Налог 1% ИП»
transactions  { date, kind: income|expense|salary, amount, accountId, categoryId,
                clientId, payerName, teacherId, comment }
```

**Главная связь.** Урок переводится в статус `conducted` → для каждого присутствующего:
`subscriptions.lessonsUsed += 1`, создаётся списание. Отменённый урок ничего не списывает.
Баланс клиента в уроках = `Σ(lessonsTotal − lessonsUsed)` по активным абонементам.
Баланс в деньгах = как сейчас, доходы минус списания.

**Питание — отдельная услуга, а не свойство пакета** (подтверждено владельцем).
Поэтому в справочнике один тариф на уроки, а питание живёт в `services` и подключается
к абонементу. При проведении урока присутствующему списывается стоимость урока
(`amountCharged`) **и отдельно** стоимость услуг (`servicesCharged`) — со своей статьёй
«Доп услуги», как это сделано в AlfaCRM (в ленте клиента видна запись
«питание остаток за прошлый период | Доп услуги | 5 000,00»).

Существующие пары пакетов из AlfaCRM при миграции разворачиваются обратно:
«Пакет 4 (1 300 000)» и «Пакет 4 без питания (1 100 000)» → один пакет на 4 урока
по 275 000 + услуга «Питание» 50 000 за занятие.

## План внедрения по фазам

Каждая фаза — самостоятельный работающий кусок, деплоится отдельно.
Порядок продуман так, чтобы справочники появились раньше, чем то, что на них ссылается.

### Фаза 0. Доступ и фундамент (полдня)
- Добавить UID специалиста в `isAllowed()` в `firestore.rules`. **Без этого он не войдёт.**
- Общий data-слой вместо трёх независимых загрузок `payments`
  (сейчас грузится в `Dashboard.jsx`, `Clients.jsx`, `Finance.jsx` целиком).
- `writeBatch` для каскадного удаления клиента (`Clients.jsx:174-180` — сейчас не атомарно).
- Переиспользовать: `src/lib/amount.js` (`toAmount`, `toCount`), стиль форм из `src/components/ClientForm.jsx`.

### Фаза 1. Справочники (1 день)
Новая страница «Настройки»: педагоги, предметы, тарифы (пакеты), кассы, статьи.

Категории расходов, сейчас захардкоженные в `Expenses.jsx:24-31`, переезжают в `categories`.

### Фаза 2. Группы (1 день)
Страница «Группы»: название, предмет, педагог, лимит мест, период, расписание, состав.
В карточке клиента — блок «Группы».

### Фаза 3. Уроки и посещаемость (2–3 дня) — **сердце системы**
- Страница «Уроки»: список + календарь на месяц.
- Создание урока: разовый и по расписанию группы.
- Журнал: отметка присутствия по каждому ученику.
- Статусы `planned → conducted | cancelled`, кнопка «Вернуть в запланированные».
- Проведение урока списывает уроки с абонементов и деньги — в одной транзакции `writeBatch`.

### Фаза 4. Абонементы (1–2 дня)
- Выдача пакета клиенту, остаток уроков, срок действия, архив.
- В карточке клиента и в списке — двойной остаток «деньги / уроки», как в AlfaCRM.
- Предупреждение, когда уроки заканчиваются (это и есть «должники к обзвону»).

### Фаза 5. Финансы (1–2 дня)
- `transactions` вместо плоских `payments`/`expenses`: кассы, статьи, плательщик.
- Тип «Выплата ЗП» с привязкой к педагогу.
- Отчёт по кассам и статьям за период. Метрики Finance пересчитать на новую модель.

### Фаза 6. Лиды (1–2 дня)
- Канбан из 6 этапов, перетаскивание карточек, архив отказов.
- Конверсия лида в клиента одной кнопкой (этап «Получена оплата»).

### Фаза 7. Миграция данных из AlfaCRM (1 день)
- Выгрузить «Экспорт в файл» по клиентам, платежам, урокам, лидам.
- Скрипт импорта в Firestore со сверкой контрольных сумм: 86 клиентов, 1461 операция.
- Прогон на копии, затем на боевой базе. Бэкап через существующий `src/lib/backup.js` до начала.

### Фаза 8. Отчёты и роли (1–2 дня)

## Проверка

Каждая фаза проверяется вручную на боевом сайте `https://crmbmk-d6303.web.app`
(локальный `npm run dev` ходит в ту же базу — тестовые записи придётся чистить).

Сквозной сценарий после Фазы 4, повторяющий реальную работу школы:

1. Завести клиента с мамой-плательщиком.
2. Выдать «Пакет 8» — остаток становится «2 400 000 / 8 уроков».
3. Создать группу с расписанием, добавить туда клиента.
4. Создать урок на дату, отметить присутствие, провести урок.
5. Убедиться: остаток стал «7 уроков», деньги списались, в ленте клиента появилась запись.
6. Отменить урок — убедиться, что списание откатилось.

После Фазы 7 — сверить: число клиентов 86, сумма операций совпадает с отчётом AlfaCRM.

Регресс после каждой фазы: `npm run build`, `npx eslint src`, вход на сайт, открытие всех вкладок.

Деплой: `VITE_BASE=/ npm run build && npx firebase deploy --only hosting --project crmbmk-d6303`,
правила — `npx firebase deploy --only firestore:rules --project crmbmk-d6303`.
