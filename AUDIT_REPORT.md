# Аудит FinGam CRM

**Дата:** 2026-05-14
**Объём:** полный аудит — стык клиент↔Firebase, фронтенд, конфигурация Firebase, архитектура, UX.
**Топология проекта:** React 19 + Vite SPA, Firebase (Auth + Firestore) как BaaS. Своего сервера нет. Хостинг — GitHub Pages. Один пользователь, в планах — мультипользовательский режим и серверная часть.

---

## 1. Executive Summary

Проект работает и решает свою задачу, код для своего размера аккуратный — нет тяжёлых зависимостей, нет грязных хаков, комментарии по делу. Но есть **три критические проблемы, которые надо закрыть в ближайшие дни**, и одна из них — прямо сегодня.

**Что горит:**

1. **База клиентов технически доступна посторонним.** Правила Firestore открыты для любого авторизованного пользователя, а Firebase позволяет постороннему зарегистрироваться через публичный API. Это значит, что персональные данные детей (имена, возраст, телефоны родителей) и вся финансовая история сейчас могут быть прочитаны или удалены кем угодно, кто откроет сайт и инструменты разработчика. Фикс — 30 минут.

2. **Критическая уязвимость в зависимости.** В составе Firebase едет пакет `protobufjs` с уязвимостью уровня critical. Фикс — 5 минут (`npm audit fix`).

3. **Нет серверной валидации данных.** Один битый документ платежа (например, с нечисловой суммой) ломает всю финансовую аналитику в `NaN`, и чинится только вручную в консоли Firebase.

**Что в хорошем состоянии:** секреты не утекли в git, CI настроен через GitHub Secrets, нет user-enumeration на логине, защита от двойного сабмита форм, нативная работа с датами без тяжёлых библиотек. Архитектурных катастроф нет — проблемы точечные и чинятся за часы, не за недели.

**Главный вывод по приоритетам:** сначала — три критические проблемы безопасности (день работы суммарно). Затем — рефакторинг в общие модули (`src/lib/`, `src/hooks/`), потому что сейчас ключевая бизнес-логика скопирована по 4 файлам и **уже начала рассинхронизироваться**. Этот рефакторинг надо сделать **до** подключения серверной части — иначе рассинхрон уедет ещё и на бэкенд. И только потом — продуктовые фичи, которые превратят «электронную тетрадь» в полноценную CRM.

## 2. Критические уязвимости и баги

### 🔴 C-1. Firestore rules открыты — утечка персональных данных детей
**Файл:** `firestore.rules:5`
**Суть:** `allow read, write: if request.auth != null` даёт полный доступ любому аутентифицированному пользователю. Регистрация через Firebase Auth REST API публично доступна (apiKey в бандле — это норма, защита должна быть в rules).
**Последствие:** утечка ПД несовершеннолетних, возможность удаления всей базы. Публичные Firebase-проекты с открытыми rules сканируются ботами массово.
**Фикс:** ограничить доступ по whitelist UID владельца. **~30 минут.**

### 🔴 C-2. Критическая уязвимость в `protobufjs`
**Файл:** `package.json` → транзитивно через `firebase ^12.10.0`
**Суть:** `npm audit` — critical (arbitrary code execution, prototype injection). Пакет попадает в прод-бандл.
**Фикс:** `npm audit fix`, затем проверить логин и загрузку данных. **~5 минут.**

### 🔴 C-3. Нет серверной валидации данных
**Файлы:** `firestore.rules` (валидация отсутствует), `Clients.jsx:115,135`, `Expenses.jsx:66`
**Суть:** rules не проверяют схему/типы/обязательность полей. `Number(f.amount)` от мусора даёт `NaN`, который уходит в Firestore и заражает все расчёты.
**Последствие:** один битый документ → вся аналитика показывает `NaN`, чинится только вручную.
**Фикс:** клиентская проверка `Number.isFinite()` + валидация схемы в rules. **~1-2 часа.**

## 4. Roadmap фич на 1-3 месяца

### Месяц 1 — CRM начинает «подсказывать»
- **Дашборд «на сегодня»** — вместо статики за всё время: блоки «Должники (звонить)», «У кого заканчиваются занятия», «Платежи за неделю».
- **Счётчик «занятий осталось»** — бейдж на карточке клиента, предупреждение когда пакет кончается.
- **Полезные пустые состояния** + сортировка должников наверх.

### Месяц 2 — Отчётность и подготовка к мультипользователю
- **Экспорт в Excel/PDF** — выгрузка платежей/расходов/прибыли за период (SheetJS, на клиенте).
- **История изменений (аудит-лог)** — коллекция `audit`: кто/что/когда. Критично *до* подключения сотрудников.
- **PWA** — манифест + service worker, установка на телефон как приложение.

### Месяц 3 — Серверная часть и удержание клиентов
- **Cloud Functions** — первый реальный кейс: предрассчитанные агрегаты (снимает H-1), серверная бизнес-логика.
- **Напоминания родителям** — Telegram-бот / WhatsApp: «осталось 1 занятие», «задолженность». Главный рычаг удержания клиентов школы.
- **Мультипользовательский режим** — коллекция `users` с ролями, обновление `firestore.rules` под роли.

## 5. Метрики, которые стоит начать трекать

**Бизнес (здоровье школы):**
- Сумма долга клиентов и число должников — в динамике по неделям.
- Средний срок погашения долга (дни от возникновения до оплаты).
- Отток: клиенты без занятий N недель.
- % клиентов, ушедших с положительным балансом (недозанимались — недополучил прибыль).
- % продлений пакета до того, как занятия кончились.
- Средний чек, выручка/расходы/прибыль помесячно.

**Продукт (используется ли CRM):**
- Время от открытия CRM до первого действия.
- Частота внесения платежей задним числом (показывает, нужна ли фича редактируемой даты — уже подтверждено).

**Технические:**
- Число чтений Firestore в день (контроль стоимости, особенно до фикса H-1).
- Размер прод-бандла (`npm run build` — пока не измерен, Firebase SDK тяжёлый).
- Ошибки в проде (после внедрения — хотя бы `console` → внешний логгер или Sentry).

## 6. Технический долг

**Копится сейчас, требует внимания до серверной части:**
- **Отсутствие общих модулей** (`lib/`, `hooks/`) — корень H-9, H-10, H-11, M-3, M-13. Каждая новая страница умножает дублирование. Чинить в Спринте 2.
- **Нет тестов** — финансовые расчёты не покрыты ничем. Станет блокером, как только логика усложнится. Решается после рефакторинга (появится что тестировать).

**Копится, можно отложить, но не забыть:**
- **Стили хардкожены** (M-14) — Tailwind подключён, но не используется; палитра размазана inline. Долг растёт линейно с каждой страницей. Чинить постепенно.
- **`Clients.jsx` 391 строка** (M-15) — первый файл, который станет неуправляемым. Разбить на компоненты.
- **Нет TypeScript** — для приложения, где половина багов это `undefined`/`NaN` в суммах, типы окупились бы. Это решение, не фикс — обсудить отдельно. Минимум — JSDoc на расчётных функциях.

**Архитектурные развилки на горизонте:**
- **GitHub Pages → Firebase Hosting** — `firebase.json` уже в репо, хостинг по сути настроен. Даст контроль над security-заголовками (M-11) и уберёт костыль с `404.html`. Обсудить при подключении серверной части.
- **Загрузка коллекций целиком** (H-1) — терпимо сейчас, бомба замедленного действия. Правильное решение — предрассчитанные агрегаты на Cloud Functions, что совпадает с планами по серверной части.
